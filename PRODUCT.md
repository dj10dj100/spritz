## The Pitch

A mobile-first leaderboard for a group of friends in Italy to track who's drinking the most Aperol spritzes. Fun, animated, adult-toned. Built to be live on real phones in days, not weeks.

## What Makes This Cool

- The **public feed** is the product. A tap turns into confetti, into a leaderboard shuffle, into a notification on everyone else's phone. The leaderboard isn't a dashboard — it's a group chat with extra steps.
- Aperol orange is doing the heavy lifting design-wise. Saturated orange + cream + bottle-green accents, big typography, animated count-ups, glass-clink sound on each tap.
- It's ephemeral. A trip-scoped competition with a start and end. There's a winner at the end. There's a loser. There's a screenshot worth keeping.

## Assumptions (override any of these and I'll adjust)

| Decision | Default I'm picking | Why |
|---|---|---|
| Group size | 4–12 people | Anything bigger and the leaderboard gets noisy |
| Trip timing | Soon (weeks, not months) | You said "produce this quickly" |
| Auth | Just pick your name from a list, no password | Zero friction, full trust, your friends aren't your threat model |
| Counting | Tap a button, count goes up | Honor system. Public feed = social accountability |
| Hosting | Vercel (free) | Push and it's live |
| Persistence | Supabase Postgres (free tier) | Free, real-time built in, no backend to write |

## Premises (push back if any feel wrong)

1. **No real auth.** A group of 4–12 friends doesn't need email signup. Anyone can be anyone. The shame of fake-tapping someone else's count is social, not technical.
2. **The feed is the feature.** A bare leaderboard would work, but it'd be boring after one round. The live feed of "Dan just clocked his 4th 🍹" is what makes people open the app at 11pm.
3. **One trip, one instance.** Don't build a multi-tenant SaaS. Hardcode the group, hardcode the dates. You can copy-paste a fresh deploy for the next trip.
4. **Mobile-only (but works on desktop).** Every interaction is designed for thumbs.

## Approaches Considered

### Approach A: Next.js + Supabase + Vercel (RECOMMENDED)
**Summary:** Server-rendered Next.js app, Supabase for Postgres + realtime channels, deploy to Vercel.
**Effort:** S (one evening to MVP, one weekend to polish)
**Risk:** Low — boring, well-trodden stack
**Pros:**
- Realtime leaderboard updates come for free with Supabase channels
- Vercel deploy is `git push`
- Free tier covers a group of 12 for a 2-week trip with massive headroom
- TypeScript everywhere — refactors are safe
**Cons:**
- Heavier than strictly necessary; Next.js is overkill for ~3 pages
- Vendor-locks you to two services (Vercel + Supabase)

### Approach B: Vanilla Vite + SQLite + Fly.io
**Summary:** Plain React + Vite frontend, tiny Hono/Express backend with SQLite, deployed as a single Fly.io machine.
**Effort:** M (longer — you write the backend)
**Risk:** Med — more moving parts, you own realtime
**Pros:**
- One repo, one binary, fully portable
- SQLite is delightful, no schema migration drama
- Costs ~ for the trip
**Cons:**
- You write your own realtime (SSE or websockets)
- You write your own auth-shim ("pick a name")
- Slower to first commit than Approach A

### Approach C: HTML + Supabase from a static page
**Summary:** Single `index.html` with `<script type="module">` calling Supabase directly. No framework, no build step. Host on GitHub Pages.
**Effort:** XS (one afternoon, no kidding)
**Risk:** Low for the MVP, Med for the animation polish
**Pros:**
- Cannot get simpler. Two files, one database, free hosting
- Forces you to ship the smallest possible thing first
- View-source friendly — fun to share the repo with friends
**Cons:**
- Polished animations (framer-motion, lottie) get awkward without a bundler
- No TypeScript safety net
- Harder to add features after the trip if it catches on

## Recommended Approach: A (Next.js + Supabase + Vercel)

Because:
- You're going to want framer-motion or similar for the confetti, count-ups, and leaderboard shuffles. That's painful without a bundler.
- Supabase realtime is genuinely two lines of code. You'd be foolish to write your own.
- Vercel preview deploys mean your friends can test on their actual phones via a link before the trip.
- If the app survives the trip and people want a "round 2 in Croatia" version, the code is reusable.

If speed-to-ship is the *only* thing that matters, drop to Approach C. But honestly Approach A only costs you ~30 extra minutes of setup.

## Data Model (Approach A)

```sql
-- One row per trip; usually you'll only have one
create table trips (
  id uuid primary key default gen_random_uuid(),
  name text not null,           -- "Italy 2026"
  starts_at date not null,
  ends_at date not null,
  created_at timestamptz default now()
);

-- One row per participant
create table participants (
  id uuid primary key default gen_random_uuid(),
  trip_id uuid references trips(id) on delete cascade,
  name text not null,            -- "Dan"
  emoji text default '🍹',       -- their avatar emoji
  color text default '#FF6B35'   -- their leaderboard color
);

-- One row per spritz
create table spritzes (
  id uuid primary key default gen_random_uuid(),
  participant_id uuid references participants(id) on delete cascade,
  consumed_at timestamptz default now(),
  location text,                 -- optional: "Piazza Navona"
  note text                      -- optional: "first of the day"
);
```

Row-Level Security: read everything, write only via a server action that doesn't need auth (or use the anon key with permissive policies — friends aren't your threat model).

## Key Screens

1. **Pick-your-name screen** — first visit, big tappable cards with each person's emoji + name. Saves to localStorage. Done.
2. **The Hub** — the home screen. Three things stacked:
   - Big animated leaderboard (top 3 with podium graphics, then the rest)
   - Massive "+1 SPRITZ 🍹" button (the main interaction)
   - Live feed below ("Dan just had his 4th • 2m ago")
3. **The Stats page** — total spritzes by the group, spritzes/hour graph, longest dry streak, "spritz of the day" award
4. **The Loser Screen** — last place gets a slightly-mocking page with rotating fake "encouragement" ("you're doing your liver a favor")

## Animation/Vibe Checklist

- Tap the +1 button → button squishes, count animates up, a single Aperol orange confetti burst, glass-clink sound
- Leaderboard re-ranks live with spring physics (framer-motion `layout`)
- Live feed slides in from the bottom, stays for a few seconds, settles
- Background is a subtle moving gradient — orange to cream
- Top of leaderboard gets a slow shimmer/glow
- 1st place gets a 👑 emoji that bounces lightly
- Empty state ("no spritzes yet") is the only place I'd put a written joke

## Distribution

- Deploy to Vercel — `git push` → live URL
- Share the URL in WhatsApp before the trip
- Add to home screen (PWA manifest) so it feels app-y on phones
- No app store. No build pipeline beyond Vercel.

## Open Questions

- Do you want a way to *undo* a tap (misclick)? Honest answer: yes, but a 10-second "undo" button on the last action is enough.
- Should one person be the "trip admin" who can edit counts, or fully no-edits-after-tap?
- Do you want a Slack/WhatsApp webhook for "Dan just hit milestone X"? Cool but probably scope-creep.

## The Assignment

Before you write a line of code: open WhatsApp, message the group, ask them to give you their **name + favorite emoji** for the leaderboard. That's your seed data. Do this before scaffolding the app — getting names live is the moment this stops being your project and becomes a shared thing.
